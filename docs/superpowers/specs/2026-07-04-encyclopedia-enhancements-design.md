# Encyclopedia Enhancements — Favorites + Language Toggle — Design

## Context

The Enciclopedia tab (added in a prior cycle) lets anyone at the table browse/search 9 categories of 5etools-derived reference data. Two gaps came up in real use:

1. No way to quickly get back to the handful of spells/rules that come up every session — you have to re-search or re-browse every time.
2. All content is in English (the `5etools-src` source language). Some players would rather read in Spanish, matching the rest of the app's Spanish UI.

This spec covers both, since they're both small, self-contained additions to `EncyclopediaTab.tsx` sharing the same persistence mechanism. Swipe-gesture tuning (a separate, unrelated request from the same conversation) gets its own spec.

**Constraint on the language toggle:** `5etools-src` has no Spanish translation of rules/spell text — its "languages" data is in-universe fictional languages (Common, Elvish, Draconic), not a localization system. There is no authoritative source to extract from. Per this project's rule that `5etools-src` is the sole source of truth for rules content, translations here are explicitly **not** rules content extracted from a source — they're a machine/LLM-generated convenience translation, clearly labeled as unofficial, scoped to the 4 categories where mid-session lookups matter most (conditions, actions, skills, spells — 442 entries). Weapons/armor/gear/mastery/feats stay English-only; their stat-block-style content is short and less prose-heavy anyway.

## 1. Persistence

Extend `AppSettings` in `src/lib/types.ts` with:

```ts
export interface AppSettings {
  theme: "piedra-viva" | "dark-fantasy";
  lastCharacterId: string;
  density: "compact" | "spacious";
  encyclopediaFavorites: string[];
  encyclopediaLanguage: "en" | "es";
}
```

- `encyclopediaFavorites`: array of `EncyclopediaItem.id` values (e.g. `"spells-Fireball"`), default `[]`.
- `encyclopediaLanguage`: default `"en"`.

No migration needed — `AppSettings` isn't versioned like `Character`; `loadSettings()` in `src/lib/storage.ts` already merges stored settings over hardcoded defaults (`{ ...defaults, ...JSON.parse(raw) }`), so old stored settings blobs missing these keys automatically get the defaults. Add both keys to the `defaults` object in `loadSettings()`.

## 2. Favorites

- Add a `"favorites"` pseudo-category as the **first** pill in `EncyclopediaTab`, before "Condiciones". Its item list is `allItems.filter((item) => favorites.includes(item.id))`, rendered with category badges exactly like cross-category search results (reusing the existing `renderRow(item, true)` path).
- Add a star toggle to every list row (`renderRow`) and to the detail modal. Tapping it adds/removes the item's `id` from `encyclopediaFavorites` and calls `saveSettings` immediately (same pattern as `useTheme.ts`'s `toggleDensity`).
- Star state reflects whether `item.id` is in the current favorites array; filled star when favorited, outline when not.
- Empty state for the Favoritos pill: "Aún no tienes favoritos — toca la estrella en cualquier entrada."

## 3. Language toggle + translations

- Add a compact "EN | ES" segmented toggle next to the search bar in `EncyclopediaTab`, bound to `encyclopediaLanguage`, persisted via `saveSettings` on change.
- New file `src/data/translations-es.ts` (hand-authored, **not** generated by `scripts/extract-5etools.ts` — there's no Spanish source to extract), exporting:
  ```ts
  export const CONDITIONS_ES: Record<string, string>; // 15 entries, keyed by Condition.name
  export const ACTIONS_ES: Record<string, string>; // 18 entries, keyed by ActionData.name
  export const SKILLS_ES: Record<string, string>; // 18 entries, keyed by SkillData.name
  export const SPELLS_ES: Record<string, string>; // 391 entries, keyed by SpellData.name
  ```
  Each value is the Spanish translation of that entry's `description` field only — not names, not spell metadata (level/school/castingTime/range/components/duration stay as-is, since those are compact D&D notation already following the "English D&D terms" convention used throughout the app).
- **Translation style**, matching precedent already in this codebase (e.g. `migrations.ts`'s Rage feature text): general prose translates to natural Spanish; ability scores become their existing Spanish abbreviations (FUE/DES/CON/INT/SAB/CAR, via the same mapping as `abilityLabel()`); specific D&D proper-noun terms (action types like "Bonus Action", area shapes like "Sphere", named conditions, item/spell names when referenced inline) stay in English. Example:
  - EN: *"Each creature in a 20-foot-radius Sphere [Area of Effect] centered on that point makes a Dexterity saving throw, taking 8d6 Fire damage on a failed save."*
  - ES: *"Cada criatura en una Sphere [Area of Effect] de 20 pies de radio centrada en ese punto debe hacer una tirada de salvación de DES, recibiendo 8d6 de daño de Fuego si falla."*
- In the detail modal, when `encyclopediaLanguage === "es"` and the active item's category is one of the 4 translated ones: show a small disclaimer line above the description — *"Traducción no oficial — verifica en inglés si hay dudas."* — then the translated text (falling back to English with no disclaimer if a translation is somehow missing for a given name, which shouldn't happen since all 442 are covered).
- For untranslated categories (weapons/armor/gear/mastery/feats), the toggle has no visible effect — always show English, no disclaimer.
- This is a substantial one-time content-generation effort (442 entries, mostly the 391 spells) — during implementation, this is authored directly by the implementing agent following the style guide above, not something that can be fully hardcoded into an implementation plan document ahead of time given its volume.

## Out of scope

- No translation of weapon/armor/gear/mastery/feat content.
- No translation of spell/item **names** — only descriptions.
- No per-favorite ordering/sorting UI — favorites list uses the same ordering as `allItems` (category-then-declaration order).
- No changes to `Character` type or migrations — both new features live entirely in `AppSettings`.

## Verification

`npx tsc --noEmit && npm run build && npm run lint` must pass. Manually verify in the dev server / browser: star a few items across different categories, confirm they appear in the Favoritos pill with correct badges and persist across a page reload; toggle EN/ES and confirm spell/condition/action/skill detail modals show the Spanish text with disclaimer while weapon/armor/gear/mastery/feat modals stay English and unaffected.
