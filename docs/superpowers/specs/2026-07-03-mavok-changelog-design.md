# Mavok Changelog — Design Spec
Date: 2026-07-03

## Overview

A read-only "Historial de versiones" list in the Settings tab, letting the user see how the app has evolved across its development rounds. No character data involved — this is app-level history, hand-maintained alongside the code, not derived from the user's LocalStorage or from git at runtime (the static-export build has no access to git history in the browser).

---

## Section 1 — Data model

New file `src/data/changelog.ts`, hand-maintained (same maintenance model as `src/data/mavok-default.ts` — edited directly, not generated by a script):

```typescript
export interface ChangelogEntry {
  version: string;   // "v1", "v2", ... sequential, not semver
  date: string;      // ISO date (YYYY-MM-DD)
  title: string;     // short Spanish title
  summary: string[]; // bullet points, Spanish, end-user framing (not dev/commit jargon)
}

export const CHANGELOG: ChangelogEntry[] = [
  // newest first
];
```

Granularity is one entry per finished round of improvements (not per commit — this repo has 114 raw commits, most of which are internal fixes/chores not meaningful to the end user; and not per version-number bump, since `package.json`'s version field isn't bumped often enough to be a useful granularity). Seeded with 7 entries reconstructed from the actual design specs in `docs/superpowers/specs/`, newest first. **Corrected during self-review**: the Rage fire-square redesign and the "Piedra Viva" visual identity overhaul (floating island nav, stone texture, cord line, framer-motion animations, toasts) were originally drafted under v2, but cross-checking `git log` against each spec's commit shows all of that shipped *before* round 1's design spec (`mavok-improvements`) was even committed — it belongs to v1, the initial build-out, not round 1.

**Maintaining it going forward:** at the end of each future round (as part of finishing that round's work), add one new entry at the top of `CHANGELOG` summarizing that round's user-facing changes in the same style. This is a manual step for whoever runs the round-closing process — no automation is added for it in this spec, consistent with keeping wording editorial/curated rather than mechanically derived.

## Section 2 — UI: placement and rendering

**File:** `src/components/tabs/SettingsTab.tsx`

- New `CollapsibleSection title="Historial de versiones"`, placed directly above the existing "About" footer block (`Mavok PWA` / character name / `D&D 5.5e (2024) · Data v{CURRENT_DATA_VERSION}` / commit SHA) — both are app-level metadata, so they sit together, separate from the character-specific "Historial de niveles" section earlier in the same tab (that one is Mavok's own level-up log; this one is the app's).
- Renders `CHANGELOG.map(...)`, newest first (array order, no sorting needed since the data is already authored newest-first).
- Each entry is a tappable row showing `version` + `title` + `date` when collapsed (e.g. "v6 · Mecánicas de Bárbaro · 02/07/2026"). **Do NOT format the date via `new Date(entry.date).toLocaleDateString("es")`** — that's the convention used elsewhere in this file (level-up history, backups), but those store full ISO *timestamps*, while `ChangelogEntry.date` is a plain `YYYY-MM-DD` string. Routing a date-only string through `new Date(...)` parses it as UTC midnight; `toLocaleDateString` then renders it in the browser's local timezone, which can display the wrong day in any timezone behind UTC (e.g. Argentina, UTC-3, would show "01/07/2026" for a "2026-07-02" entry). Instead, format directly from the string without constructing a `Date`:
  ```typescript
  const [year, month, day] = entry.date.split("-");
  const displayDate = `${day}/${month}/${year}`; // "2026-07-02" -> "02/07/2026"
  ```
- Tapping toggles that entry's expansion, showing its `summary` as a bullet list. Only one entry expanded at a time — same pattern as `InventoryTab.tsx`'s `expandedItem: string | null` state (tapping an already-open entry's id again collapses it; tapping a different entry switches to it). This matches the closest existing precedent for "list of records, each with an expandable detail body" in this codebase (as opposed to `CollapsibleSection`'s independent multi-open sections, which is for distinct top-level page sections, not list items).

## Section 3 — Read-only, no character interaction

No character data is read or written by this feature. It doesn't need `useCharacterContext()` at all — `CHANGELOG` is static, imported data, rendered unconditionally regardless of whether a character is loaded. This is simpler than every other section in `SettingsTab.tsx`, all of which depend on `character`.

## Explicitly Out of Scope

- Auto-generating entries from git log or from `docs/superpowers/specs/*.md` at build time — rejected in favor of hand-authored entries, so wording can be tuned for the end user rather than mechanically derived from developer-facing spec/commit text (see Section 1).
- Semantic versioning (`package.json`'s `version` field is unrelated to this feature's `version: "vN"` labels, and isn't bumped by this change).
- Any change to `CURRENT_DATA_VERSION` or data migrations — this is pure static content, no `Character` schema involvement.
